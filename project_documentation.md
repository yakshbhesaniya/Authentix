# Authentix: Comprehensive Project Documentation

## 1. Project Overview

### System Capabilities
Authentix is a powerful, production-ready Content Intelligence Platform. It features three interconnected capabilities:
1. **Plagiarism Detection**: Uses lexical shingling alongside semantic sentence embeddings (SBERT) to detect both exact copying and sophisticated paraphrasing.
2. **AI-Generated Content Detection**: Yields a calibrated probability score (0–100) indicating the likelihood a text was generated by AI models (like ChatGPT, Claude, etc.), leveraging an ensemble of Transformer classifiers, stylometric feature extraction, perplexity calculation, and linguistic distribution divergence.
3. **Humanization Engine**: Re-writes AI-detected content using a controlled generation pipeline that explicitly protects factual meaning, named entities, logical structure, and numerical data via strict semantic validation mechanisms.

### The Problem It Solves
Academic institutions, enterprises, and publishers constantly battle content authenticity. Traditional plagiarism checkers solely rely on exact keyword matches, failing entirely against texts re-written by AI. Meanwhile, basic AI detectors have high false-positive rates due to relying on a single neural network model, leading to false accusations. Furthermore, individuals trying to "humanize" text often use tools that hallucinate data or ruin intent. 

Authentix solves this by fusing deterministic text matching with probabilistic ML logic, delivering highly accurate, explainable results, and ensuring that any "humanization" is rigorously validated before being presented to the user.

### Why This Architecture Was Chosen
A microservice-based architecture was selected because Machine Learning workflows are inherently incompatible with traditional web-request architectures. 
* ML models (Python) require heavy memory allocation (loading multi-gigabyte `.bin` files into VRAM) and block CPU execution threads while multiplying matrices. 
* The API Gateway (Node.js) is designed for non-blocking I/O—meaning it can receive thousands of concurrent file uploads and handle constant client connection polling without stalling.

By tearing the ML execution away from the HTTP gateway and using an asynchronous message broker (Redis + BullMQ) in between, the frontend and API remain incredibly responsive even if background GPU tasks take minutes to complete.

### Core Feature Interactions
The system's three features are highly interdependent. For instance:
1. When generating a Full Report, a single uploaded payload is parsed once and broadcast to the Plagiarism Engine and AI Detection Engine concurrently. 
2. The Humanization Pipeline leverages the exact same SBERT similarity algorithms used in the Plagiarism Engine to validate that the rewrite hasn't drifted semantically from the source text. 
3. The Humanizer uses the AI Detection engine directly in its ranking phase; it scores generated rewrite candidates and prioritizes the one that scores highest in "Human probability."

### Target Users & Scalability Goals
* **Target Users**: Universities verifying student essays, publishers verifying freelance submissions, and enterprises enforcing content authenticity policies.
* **Scalability Goals**: The system must process files up to 100k+ words. It must scale horizontally—if AI Detection tasks bottleneck, DevOps can add 5 AI Detection containers without modifying any code or changing the Gateway, Plagiarism, or Humanizer services.

### Full Microservice Infrastructure
The infrastructure bridges high-throughput web architecture with GPU-capable ML services.

### Request Lifecycle Step-by-Step
1. **User uploads file**: A user uploads a 50-page PDF on the Next.js frontend. The client pushes a `multipart/form-data` payload to the Gateway's `/upload` endpoint.
2. **API receives request**: The Fastify Gateway verifies the user's JWT strictly. The Rate Limiter allows the request to pass.
3. **File ingestion (Storage)**: The Gateway pipes the file stream directly into MinIO (an S3-compatible local bucket) to avoid holding 50MB of data in Node.js RAM. It generates a unique `file_key`.
4. **Queue dispatch**: A new Job record is inserted into PostgreSQL with status `queued`. An event `{ jobId, type: "analyze", fileKey }` is pushed onto the Redis queue. Fastify immediately returns a `202 Accepted` to the client. The client begins polling the `/jobs/:id` endpoint.
5. **ML service processing**:
   - A background BullMQ Worker (running alongside the gateway, or decoupled) claims the job from Redis.
   - It makes an HTTP POST to the Python **Ingestion Service** (`/parse`) passing the `fileKey`. The Ingestion service downloads the file from MinIO, invokes `pdfplumber`, strips headers/footers, and returns structured JSON (paragraphs, tokens).
   - The Worker then fires two parallel HTTP requests to the **Plagiarism Service** and **AI Detection Service** with the normalized text payload. 
   - The Plagiarism Service embeds sentences and queries similarities against **Qdrant**. The AI Detection Service invokes Transformer layers.
6. **DB Storage**: The Worker receives the responses, compiles them into a single report object, and invokes an `INSERT INTO analysis_results ... ON CONFLICT DO UPDATE` query into PostgreSQL. It updates the parent Job status to `completed`.
7. **Response return**: The client polling the Gateway sees the status transition to `completed` and the database delivers the full JSON results graph.

### Why Particular Technologies Were Selected
- **Why API separated from ML service?** Node.js is single-threaded but excels at handling 10,000 idle websocket/polling connections utilizing the libuv event loop. Python's Global Interpreter Lock (GIL) and heavy synchronous processing for AI models (loading PyTorch tensors) will block process execution. Merging them creates catastrophic bottlenecks.
- **Why Redis?** Message queues require ultra-fast, in-memory locking mechanisms. Redis natively supports atomic LUA scripts, which BullMQ uses to guarantee a background job is never picked up by two workers simultaneously.
- **Why PostgreSQL?** Job status transitions and user billing records require ACID guarantees (Atomicity, Consistency, Isolation, Durability) so data cannot be lost in a split-brain scenario.
- **Why Vector DB (Qdrant)?** Comparing a submitted document's embedding vector against 100 million stored vectors via linear search would take minutes. Qdrant uses HNSW (Hierarchical Navigable Small World) graphs to find semantic nearest neighbors in sub-milliseconds.
- **Why Docker?** Native Python libraries (like PyTorch or datasketch) rely heavily on distinct OS-level C++ bindings that vary drastically across Windows, Mac, and Linux. Docker freezes the exact binary dependencies, meaning production execution perfectly mimics the developer's laptop.

### Why this Monorepo Structure?
A monorepo structure was deliberately chosen over maintaining separate repositories. A monorepo ensures that changes spanning the Gateway, Web, and Python layers happen cohesively in a single Atomic git commit.

## 4. Dependencies Explanation

### Node.js Backend (`apps/gateway/package.json`)
*   **`fastify`**: The core API framework. **Why:** It processes JSON 20% faster than Express.js due to internal schema compilation. **Without it,** we would have to write plain `http` sockets and handle complex routing structures natively.
*   **`@fastify/jwt`**: JSON Web Token generator/validator. **Why:** Necessary for stateless authentication logic ensuring horizontal scaling.
*   **`@fastify/rate-limit`**: DDoS protection. **Why:** Rejects API calls when limit exceeded (HTTP 429), preventing our expensive GPU microservices from being overwhelmed by robotic traffic.
*   **`@fastify/multipart`**: Payload buffer streaming. **Why:** Connects to Node's stream API allowing 50MB files to stream directly to S3 without loading strictly into RAM.
*   **`bullmq` & `ioredis`**: Robust background job management. **What:** Coordinates queue distribution on top of Redis. **Without it,** background jobs would crash silently if a server restarted mid-process.
*   **`pg` & `drizzle-orm`**: The PostgreSQL client and strict TypeScript Object Relational Mapping layer. **Why:** Converts SQL tables to typed interfaces. Drizzle avoids the huge binary overhead of Prisma, providing lean, rapid database querying.
*   **`zod`**: Schema enforcement logic. **Why:** Strips bad data off inbound API requests instantly. **Without it,** malformed strings could propagate down to python models and cause critical unhandled exceptions.
*   **`minio`**: The AWS S3 compatible client. **Why:** Moves binary files to storage automatically saving our DB from bloat.
*   **`bcryptjs`**: Cryptography. **Why:** Securely hashes passwords using the blowfish algorithm. 

### Python ML Services (`services/*/requirements.txt`)
*   **`fastapi`**: Python ASGI framework. **Why:** Asynchronous handling of HTTP routes.
*   **`uvicorn[standard]`**: The ASGI web server. **Why:** Runs the fastAPI logic. Python can't natively parse HTTP requests over websockets efficiently without an ASGI server handling the translation.
*   **`sentence-transformers`**: HuggingFace library. **Why:** Loads dense vector models (e.g., SBERT) converting English paragraphs into numerical arrays of floats representing contextual meaning. **Without it,** the plagiarism engine could only see exact text overlaps, utterly failing against paraphrasing.
*   **`qdrant-client`**: Database client. **Why:** Required to interact with the Qdrant instance storing millions of embeddings natively.
*   **`datasketch`**: Algorithm implementation library. **Why:** Manages MinHash and LSH calculations enabling O(1) comparison speeds over gigantic datasets for exact-overlap detection without database bottlenecks.
*   **`numpy` & `scipy`**: Low level C-bindings for multidimensional arrays. **Why:** Handling massive matrices.
*   **`pydantic`**: Strict data typing for Python classes. **Why:** Enforces input/output contracts so our REST calls from Node don't break. 
*   **`spacy`**: Industrial NLP pipeline. **Why:** Performs Named Entity Recognition (finding "John", "January", "Google"). Used exclusively by the Humanization firewall to ensure the rewrite didn't delete real-world facts.

### 5.1 The Gateway Entry Point (`apps/gateway/src/server.ts`)

```typescript
import fastify from "fastify";
import jwt from "@fastify/jwt";
import cors from "@fastify/cors";
import rateLimit from "@fastify/rate-limit";
import multipart from "@fastify/multipart";
// 1. IMPORT DEPENDENCIES: We import the framework and all associated security/routing plugins. 
// If these fallbacks are missing, the server is insecure (no cors) or open to DDoS (no rateLimit).

import { getDb } from "./db/client";
import { authRoutes } from "./routes/auth";
import { jobRoutes } from "./routes/jobs";
import { startWorker } from "./queue/worker";
// 2. INTERNAL MODULES: Bring in our database connection singleton, separated API routes, and BullMQ worker.

const app = fastify({ logger: true });
// 3. INIT: Create the web server instance. enabling { logger: true } binds Pino to the request context.
// Why: Production debugging relies entirely on logged JSON trails to trace bugs.

app.register(cors, { origin: "*" });
// 4. CORS: We instruct the server to accept traffic coming from outside domains (e.g., our Next.js UI).
// Alternative: If restricting to enterprise usage, we would bind { origin: "https://authentix.app" }

app.register(jwt, { secret: process.env.JWT_SECRET || "supersecret" });
// 5. JWT: Registers signing configuration. In prod, process.env guarantees keys aren't pushed to Git.

app.register(multipart, { limits: { fileSize: 50 * 1024 * 1024 } });
// 6. UPLOADS: Bounds memory intake. If we removed the limits block, an attacker could stream a 300GB file 
// leading to a total crash of the node process due to OOM (Out Of Memory) exceptions.

app.register(rateLimit, { max: 100, timeWindow: "1 minute" });
// 7. THROTTLING: Prevents more than 100 requests per IP address automatically.

app.register(authRoutes, { prefix: "/auth" });
app.register(jobRoutes, { prefix: "/jobs" });
// 8. MOUNTING: Attaches our encapsulated route functions mapped natively behind the /auth and /jobs URIs.

app.get("/health", async () => { return { status: "ok" }; });
// 9. LIVENESS PROBE: Kubernetes, Docker Swarm, and AWS ping this URL to determine if the container crashed.

const start = async () => {
    try {
        await getDb().connect();
        // 10. CONNECT OR DIE: If the PG database is down, it halts booting. We shouldn't serve traffic 
        // if we can't save the results.
        
        await startWorker();
        // 11. QUEUE: Starts actively polling Redis to pick up and process ML jobs asynchronously.
        
        await app.listen({ port: 8000, host: "0.0.0.0" });
        // 12. BIND PORT: Tell Node to listen on traffic. 0.0.0.0 exposes it globally out of the Docker sandbox.
        // Alternative: binding to 127.0.0.1 (localhost) would render it completely unreachable externally.
        
        console.log("Gateway listening on port 8000");
    } catch (err) {
        app.log.error(err);
        process.exit(1);
        // 13. FAIL SAFELY: Instead of freezing, we force an exit code 1.
    }
};
start();
// 14. INITIATE LIFECYCLE: Fire the asynchronous execution sequence at the root.
```

## 6. Docker Deep Explanation

Docker completely eradicates the "it works on my machine" problem.

- **What is a Docker image?** It's a static blueprint containing down to the binary level the OS, drivers, code, and package versions required.
- **What is a Container?** An active instance forged from the image. It runs isolated, seeing its own filesystem entirely divorced from your Windows/Mac host.
- **What is a Layer?** Dockerfiles execute step by step. Each command pushes a caching "Layer" to disk.

### The Gateway Dockerfile (`apps/gateway/Dockerfile`)
```dockerfile
FROM node:20-alpine
# 1. BASE: We pull Alpine linux (5MB image) instead of standard Ubuntu (80MB).

WORKDIR /app
# 2. CONTEXT: Switches context inside the miniature Linux machine.

COPY package.json package-lock.json ./
# 3. CACHE PREP: Copy purely the requirement lists first.

RUN npm install
# 4. DEPENDENCIES: Install. Why split this from copying source code? Docker caches layers. 
# If we edit `server.ts` later, Docker WILL NOT run `npm install` again, grabbing the cached layer instantly.

COPY . .
# 5. BUNDLING: Drag the entire folder contents recursively into the container.

RUN npx tsc
# 6. COMPILATION: Translate TypeScript to JavaScript, producing the `./dist/` folder inside the container.

EXPOSE 8000
# 7. DOCUMENTATION: pure meta-tag indicating intent.

CMD ["node", "dist/server.js"]
# 8. EXECUTION PROCESS: Specifies the parent PID 1 command sequence.
```

### Understanding `docker-compose.yml`
```yaml
services:
  gateway:
    build: 
      context: ./apps/gateway
    ports:
      - "8000:8000"
    depends_on:
      - postgres
      - redis
    environment:
```

- **`services`**: We define all subcomponents of our architecture here natively.
- **`build.context`**: Evaluates exactly where the specific Dockerfile lies in relation to the root.
- **`ports`**: The string "8000:8000" maps your physical machine's `localhost:8000` network card interface straight through the container wall directly to the isolated instance's internal `8000` port.
- **`depends_on`**: Guarantees boot order.
- **`environment` & DNS**: `postgresql://...` notice the host is typed explicitly as `postgres`. Because Docker Compose manages its own Subnet IP Address network natively, you don't type IPs (like `192.168.1.5`). You use the exact service name defined within the `.yml` file.

## 7. Data Flow and Processing Pipelines

### Plagiarism Pipeline Execution Flow
1. **Text Normalization:** Extraneous formatting (tabs, weird unicodes, invisible PDFs metadata) are stripped to generate a pure `utf-8` corpus string.
2. **Shingling Execution:** We chunk the unified text. If a sentence is "The quick brown fox jumps", moving 3-word shingles are arrayed: `["The quick brown", "quick brown fox", "brown fox jumps"]`.
3. **MinHash Extraction:** Hashing algorithms compress millions of shingles into 256-integer sets.
4. **Embedding Generation:** Concurrent to MinHash, structural sentences are fed into HuggingFace `all-MiniLM-L6-v2`. Semantic float arrays are yielded.
5. **Vector Similarity:** We request Qdrant to search its spatial mapping grid to return vectors located inside bounding thresholds > 94% spatial closeness.
6. **Scoring Aggregation:** LSH overlaps yield an exact lexical score. Vector distance yields the semantic score. An algebraic scaling calculation blends them into the true final %.

### AI Detection Pipeline Flow
1. **Perplexity Computation:** A GPT-2 instantiation checks word tokenization predictability. "The sky is" strongly expects "blue" (high probability, low perplexity). AI generation is incredibly low perplexity due to probability-driven math paths.
2. **Stylometric Feature Extraction:** Iterators traverse sentences analyzing variance gaps in word counts matching specific human grammatical syntax distributions.
3. **Transformer Classifier:** `RoBERTa-base` runs sequential attention mechanism layers calculating contextual weights against trained datasets explicitly modeled over LLM generation artifacts.
4. **Ensemble Stacking:** We feed the numerical outputs of steps 1, 2, and 3 straight into a Scikit-Learn `LogisticRegression` algorithm pre-calibrated explicitly under Platt Scaling arrays to return an actual % probability format output.

### Humanization Pipeline Validation Logic
1. **Parsing:** Extracts sentences.
2. **Controlled Generation:** `T5-base` uses multi-sequence decoding. A beam sample produces 5 iterations. We adjust `temperature=0.8` internally to explicitly force the model to occasionally choose suboptimal, non-standard structural paths (humanizing the text).
3. **Semantic Validation:** We map both the original string AND rewritten output string back through `SBERT`. The angular Cosine Distance calculates strictly bounded semantic consistency. If `SBERT(Input) < 0.92 SBERT(Output)` -> Drop candidate instantly.
4. **Entity Lock Check:** spaCy verifies Named Entities. If the original had "2023" and "Japan", and the candidate says "Next year" and "Asia", it fails the validation lock and re-tries immediately.
5. **Similarity Threshold Validation:** Returns whichever valid remaining candidate scores lowest locally on the internal AI Detector test function.

## 8. Scalability and Production Readiness

- **Horizontal Scaling:** We scale by replicating individual Docker containers. If Plagiarism detection is slow, we spin up 5 identical `plagiarism:latest` instances. The Gateway routes traffic evenly between them.
- **Stateless Services:** Absolutely no context variables, DB saves, or session states are held in Fastify RAM or FastAPI RAM. You can delete or restart a container mid-operation, and the BullMQ broker (living firmly in Redis) simply observes the timeout and shifts the active execution payload back to the available queue logic stack.
- **Load Balancing:** Utilizing Nginx Reverse Proxy implementations handles automated round-robin routing directly across identical container subsets.
- **Caching & Embedding Retention:** Rather than re-requesting SBERT embeddings for duplicate or overlapping database records, identical file hash lookups are maintained locally preventing costly GPU matrix recalibrations.
- **Background Processing Constraints:** API interactions shouldn't ever exceed 400ms. By using workers to handle tasks needing upwards of 12 seconds, Node continues serving thousands of users unencumbered.
- **Cold Starts Avoidance:** We bypass serverless environments in favor of container orchestration so `.bin` AI models remain booted natively inside python Virtual Memory rather than forcing 3GB RAM initialization penalties per request.

## 10. Security Considerations

- **File Validation:** MinIO streams automatically refuse execution parameters. `multipart` guarantees file-name extension truncation and MIME-type restrictions preventing shell executions (.sh/.exe).
- **Rate Limiting (API Throttling):** Fastify executes algorithmic token bucketing explicitly locking API requests blocking robotic scraping queries immediately.
- **Authentication:** Application of JWT verifies explicit identity definitions via HMAC SHA-256 verifying cryptographic claims on stateful interactions mapping precisely back to bounded user ids exclusively.
- **Input Sanitization:** Pydantic evaluates ALL Python dictionaries blocking injection attacks explicitly while PostgreSQL driver abstractions execute explicit parameterized query bindings preventing all potential SQL Injections outright.
- **Model Abuse Prevention:** Limiting maximum API text arrays to 50k constraints natively preventing intentional algorithmic looping attacks constructed by malicious parties trying to cause infrastructure resource exhaustion mechanisms.
